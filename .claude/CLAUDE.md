# 基本的なルール

- タスクの事前調査にはサブエージェントを使用する
- プロダクトコードを作成・変更した場合は、次のような確認を行い問題が発生していないことを確認する。テストコード、ドキュメントは除く
  - ファイルの修正後に、`pnpm prettier --write <file>` と `pnpm eslint --fix <file>` を同時に実行しファイルのフォーマットを行う。例えば import 順序の修正はこれで簡単に行える
  - `pnpm test:run` を実行し、修正したファイルを個別にテスト実行する。テストケースを修正すべきか判断に迷った場合は確認する
  - `pnpm lint` を実行し、修正したファイルを個別に lint チェックを行う。問題があれば修正する
  - `pnpm tsc` を実行し、TypeScriptのチェックを行う。問題があれば修正する
- テストコードの作成・変更をする場合はスキルを使用する
- コミットをする場合はスキルを使用する
- .ts、.tsx のファイルには適切な TSDoc を記述する
- 共通化について以下のドキュメントを参照してください。同じようなコンポーネント・機能がすでに存在する場合は、それらを使用するようにしてください
  - @docs/components-reference.md
  - @docs/lib-utils-reference.md

## デザインコンセプト: Refined Minimalism with Subtle Interaction

プロジェクト全体の UI デザインはシンプルで落ち着いたデザイン原則に従います。

### 基本原則

1. **エフェクト最小化**
   - グローやグラデーション背景エフェクト（`FlowingGlow`、`bg-linear-to-br` など）を廃止
   - 派手なシャドウ（`shadow-lg`、`hover:shadow-xl`）を削除
   - 視覚的な装飾を極力排除し、機能と情報の伝達を最優先

2. **ホバーインタラクション**
   - 複数のアニメーション（`scale`、`translate`、`rotate`）を廃止
   - 背景色の微妙な変化のみで相互作用性を表現（例：`hover:bg-muted/50`）
   - `transition-colors duration-150` で落ち着いた変化を実現
   - ホバーした際の視覚的フィードバックは控えめに

3. **アニメーション戦略**
   - 一般的な UI 操作時のアニメーション（フェード、スライド、スケール）は廃止
   - **機能的フィードバック**: 機能の実行結果を視覚的にわかりやすくするためのアニメーション（例：ランダム生成時のスピン、正解時の色変化）は保持

4. **空間デザイン**
   - 背景：`bg-background` または `bg-muted/30` などの控えめな色
   - **枠線は極力控える**：`border-border/30` などの微妙な枠線よりも、余白と背景色の違いで境界を表現する
   - `backdrop-blur-sm` などの透視効果は廃止
   - 十分な余白でコンポーネント間の関係を示す

5. **色彩戦略**
   - ベースカラー：白黒で構成
   - アクセントカラー：amber、violet、emerald などを機能別に使い分け
   - 各コンポーネントはアクセント色でホバー時の背景が変わる

### 適用範囲

- 新規コンポーネント作成時のデフォルト設計方針
- 既存コンポーネントの UI 改善時に適用
